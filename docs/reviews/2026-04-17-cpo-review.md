# CPO Review: 모바일(iOS/Android) 출시를 위한 UX/UI 아키텍처 갭 분석

작성일: 2026-04-17
대상 빌드: `http://localhost:8765` (Expo / React Native Web)
리뷰어 관점: Senior Chief Product Officer

---

## 0. 한 줄 요지 (TL;DR)

지금 빌드는 **"케어 저니 트래커"라는 강력한 제품 테제**의 조각을 보여주지만, 본질적으로 *홈 탭 하나*만 완성된 프로토타입이다. 모바일 출시까지 가려면 **IA 완성, 플랫폼 네이티브 역량(푸시·위젯·생체인증), 동기화/권한 모델, 프라이버시·규제 대응, 디자인 시스템 토큰화** — 이 다섯 축을 병렬로 끌고 가야 한다. 카테고리(의료/건강) 특성상 *"기능 많음"보다 신뢰와 지속 사용성*이 KPI 중심이어야 한다.

---

## 1. 제품 이해 — 무엇을 본 것인가

### 1.1 핵심 제품 테제
화면 좌상단 **"에셋 전환"** 드롭다운이 전체 제품 구조를 결정한다. 한 명의 주돌봄자(primary caregiver)가 여러 *케어 에셋*을 동시에 관리한다:

| 에셋 | 맥락 | 히어로 위젯 | 하단 탭 구성 | 테마 컬러 |
|---|---|---|---|---|
| 시험관 3회차 | 본인 난임 치료 | 배아 이식 D-1, CHA 강남 | 홈·달력·감정·파트너 | 피치 |
| 보리 관절관리 | 반려견(포메, 9살) | 관절약 22:00 (2h 후) | 홈·달력·보리·설정 | 핑크 |
| 어머니 항암 | 부모 항암 케어 | 항암 외래 13:30, 서울대병원 | 홈·체크·인사이트·공유 | 슬레이트 |

같은 "시간-기반 의료 루틴"이라는 추상화 위에서 **UI 토큰·레이아웃·인터랙션이 맥락별로 적응(morph)** 되는 구조다. 이건 매우 야심 찬 포지셔닝이다 — Flo, CareZone, 11pets 같은 버티컬 앱을 **하나의 OS로 대체하겠다**는 선언에 가깝다.

### 1.2 현재 완성도
- **홈 탭만 실제로 구현됨.** 달력/감정/파트너/체크/인사이트/공유/보리/설정 등 나머지 탭은 모두 *"이 탭은 다음 단계에서 채워질 예정이에요"* 플레이스홀더.
- 온보딩, 로그인, 설정, 권한, 빈 상태, 에러 상태, 알림 UI, 입력 플로우 전부 **부재**.
- 기술 스택은 **Expo (React Native Web)**. iOS/Android 빌드 파이프라인은 이미 잠재적으로 확보됨 — 이 결정이 전략적으로 옳다고 본다.

## 2. 지금 빌드의 UX 강점 (유지해야 할 것)

1. **히어로 카드가 감정의 앵커.** D-day + 시간 + 장소가 한 장에 압축된 카드는 의료 앱의 스트레스 상황에서 **인지 부하를 극적으로 낮춘다**. 이건 락스크린 위젯과 Live Activity의 지표가 될 수 있는 가장 좋은 요소다.
2. **시간 순서의 주사/약 타임라인.** *"완료 → 지금 → 예정"* 상태 뱃지는 복잡한 프로토콜을 바로 읽히게 만든다.
3. **맥락별 하단 탭 리셰이프.** 항암은 "체크/인사이트/공유"가 말이 되고, 반려견은 "설정"이 말이 되고, 난임은 "감정/파트너"가 말이 된다 — 이건 한 앱이 세 버티컬을 먹겠다는 전략을 성립시키는 핵심 디자인 결정.
4. **감정 체크인의 존재.** 의료 앱에서 *정량*뿐 아니라 *정성* 지표를 트래킹한다는 건 차별화 포인트이자 리텐션 드라이버.
5. **수의사 메모 / 지난 회차 메모 / 진료 시 물어볼 것** 같은 의료 맥락 특화 카드는 경쟁사 대비 "의사와의 대화를 보조하는 툴"이라는 포지션을 만든다.

## 3. 모바일 출시까지 필요한 UX/UI 아키텍처 — 6가지 축

### 3.1 정보구조 (IA) 완성

현재 12개 탭 중 **75%가 빈 껍데기**. 아래 최소 세트가 탭마다 필요:

- **달력 탭**: 월/주/리스트 뷰, 항목 상세, 드래그-to-reschedule, 예정 vs 완료 필터.
- **감정 탭 (IVF)**: 히스토리 히트맵, 상관관계 뷰(약 vs 감정), 노트 입력, 파트너 공유 토글 per 항목.
- **파트너 탭 (IVF)**: 초대 플로우, 동기화 스코프 설정, 파트너에게 보이는 프리뷰, 파트너 알림 취향.
- **체크 탭 (항암)**: 질문 템플릿 라이브러리, 체크 이력, 진료일에 PDF 내보내기.
- **인사이트 탭 (항암)**: 주기별 부작용 그래프, 약 변경 이력, 영양/수면 정렬.
- **공유 탭 (항암)**: 형제/자매 공동 돌봄 초대, 권한 레이어(읽기/쓰기/알림만), 공유 로그.
- **보리 탭 (펫)**: 프로필, 의료 히스토리, 사료·체중·X-ray 기록.
- **설정**: 계정, 알림, 에셋 관리(순서·삭제·테마), 데이터 내보내기, 프라이버시 대시보드.

또한 **전역 IA에 빠진 것**:
- 앱 진입 온보딩 (사용자 본인 / 파트너 / 돌봄 대상 관계 설정).
- 에셋 생성 플로우 ("새 에셋 만들기" 버튼은 있으나 플로우 미구현).
- 로그인 / 소셜 / 매직 링크.
- 알림 인박스 (여러 에셋의 알림을 모으는 허브).
- 검색 (약 이름·증상·과거 메모).
- 프로필 전환 (소유자 본인 vs 어머니 계정 관점).

### 3.2 플랫폼 네이티브 역량 (iOS / Android가 기본 기대하는 것)

의료/케어 카테고리는 아래 요소가 **고급 기능이 아니라 테이블 스테이크**다:

| 역량 | iOS | Android | 이 제품에서의 이유 |
|---|---|---|---|
| 푸시 알림 + 로컬 리마인더 | APNs | FCM | 주사/약 타임에 **절대** 놓치면 안 됨. 반복·스누즈·조건부(식전/식후) 필요 |
| 락스크린 위젯 | WidgetKit | Glance API | 히어로 D-day 카드는 앱을 안 열어도 보여야 함 |
| Live Activity / Always-On | Dynamic Island | Ongoing Activity | "다음 주사까지 2시간" 같은 카운트다운이 앱 밖에서도 살아있어야 함 |
| 생체 인증 | Face ID / Touch ID | BiometricPrompt | 난임·항암 데이터는 극도로 민감. 앱 진입 시 기본 ON 옵션 |
| 헬스킷 / 헬스 커넥트 | HealthKit | Health Connect | 월경 주기·수면·심박 → 난임/항암 부작용 상관관계 |
| 캘린더 연동 | EventKit | CalendarContract | 병원 예약을 iOS 캘린더/구글 캘린더에 자동 추가 |
| 웨어러블 동반앱 | Apple Watch | Wear OS | 주사 완료 한 번 탭으로 기록 |
| 사진/PDF 인제스트 | PhotoKit, VisionKit OCR | CameraX, ML Kit | 진단서·처방전·X-ray 업로드 |
| 공유 시트 | UIActivity | Android Share | 의사와 PDF 리포트 공유 |
| 심층 링크 | Universal Links | App Links | 파트너 초대, 알림 탭 → 해당 에셋 |

이 중 **푸시, 위젯, 생체인증, 오프라인 로깅**은 v1 필수다. 나머지는 v1.x 우선순위로 분류할 수 있다.

### 3.3 데이터 / 계정 / 권한 아키텍처

현재 빌드는 *단일 디바이스, 로컬 상태*로 보인다. 모바일 출시 조건:

- **멀티디바이스 동기화**: 본인이 폰 + 아이패드 + 웹을 오가며 로깅할 가능성 높음.
- **에셋 소유권 모델**:
  - *Self-asset* (본인 IVF): 소유자 = 본인. 파트너는 읽기/제한적 쓰기.
  - *Surrogate-asset* (어머니 항암): 소유자 = 어머니(원론적), 돌봄자 = 본인 + 형제. **권한 행렬**이 중요.
  - *Pet-asset* (보리): 소유자 = 가족. 수의사/펫시터 공유 링크.
- **오프라인 우선 로깅**: 병원은 전파 약함. 로컬 큐 → 이후 싱크. CRDT 또는 timestamp-based merge.
- **암호화**: at-rest 필수, 파트너 동기화는 가능하면 E2EE 또는 최소한 필드-레벨 암호화.
- **데이터 내보내기 / 삭제**: GDPR, 국내 개인정보보호법(PIPA) 대응 의무.

### 3.4 프라이버시 / 규제 / 앱스토어 대응

이게 **가장 과소평가되기 쉬운 축**이다. 의료/가임 데이터는 법적·정책적 지뢰밭이다:

- **난임 데이터의 민감도**: 2022년 이후 미국 시장 진출을 고려하면 특히 취약 (Dobbs 판결 이후 리프로덕티브 데이터 규제 강화).
- **앱스토어 정책**:
  - Apple: Medical App Guidelines 1.4.1 — *진단·치료 주장 금지*, 의료 디스클레이머 필수.
  - Google Play: Sensitive Health Data 정책 — 공유/광고 제한.
  - 양쪽 다 Privacy Nutrition Labels / Data Safety 섹션 상세 기입.
- **국내**:
  - 개인정보보호법 + 의료법 — 의료기기 소프트웨어 여부 판정 필요(현재 구성은 **일반 SW 추정**, 단 "복약 알림"을 의료행위 보조로 해석할 여지 있어 식약처 자문 권장).
  - 청소년 사용자 가능성 (만 14세 미만 난임은 없지만 펫/가족 케어는 있을 수 있음).
- **파트너 공유의 윤리**: *동의 없이 본인 감정 데이터가 파트너에게 보이면 안 됨* — per-field 공유 토글과 취소 가능성이 UX의 핵심.

### 3.5 디자인 시스템 성숙도

현재 빌드는 아트디렉션은 훌륭하지만 **토큰화되어 있지 않은 것으로 보인다**. 모바일 출시 전에 필요한 것:

- **토큰**: 스페이싱(4/8/12/16/24/32), 타이포 램프(XS–XXL), 시멘틱 컬러(surface, primary, danger, success) × 에셋 테마(peach/pink/slate).
- **다크 모드**: 병원 복도·새벽 주사 시나리오에서 필수.
- **Dynamic Type / 폰트 스케일링**: 항암 보호자는 *부모님 세대가 직접 볼 수도 있다*. iOS Dynamic Type, Android fontScale 지원.
- **컴포넌트 라이브러리**: Hero Card, Timeline Row, Status Badge, Emotion Chip, Asset Card, Prompt Check Item, Mini-Calendar, Pill Card, Persona Header, Treatment Stepper — 최소 10개 패턴.
- **모션 언어**: 에셋 전환 시 컬러 모프 + 햅틱. 이게 제품의 *"와우 모먼트"*.
- **접근성 베이스라인**: WCAG AA 대비비, VoiceOver/TalkBack 레이블, 44pt 최소 타겟, 원핸드 도달성.

### 3.6 신뢰·행동 유도 UX

의료 앱의 KPI는 결국 **"지속 로깅 습관"**이다. 이를 받치는 UX:

- **제로 마찰 로깅**: 히어로 카드 자체가 *"완료 ✓"* 탭 가능해야 함 (현재는 표시만).
- **리마인더 루프**: 놓친 주사 → 10분 후 재알림 → 1시간 후 파트너에게 에스컬레이션 (옵션).
- **스트릭 / 진전감**: 과도한 게이미피케이션은 의료 맥락에 맞지 않지만, *"5주차 완료 🌱"* 정도의 조용한 축하는 효과적.
- **"지금 뭐 해야 하지?" 원탭 답**: 앱 열면 *다음 액션 하나*가 가장 눈에 띄어야 함. 현재 히어로 카드는 그 역할을 하고 있음 — 유지.

## 4. 경쟁 환경과 포지셔닝 리스크

솔직한 CPO의 말: **"한 앱이 난임·항암·펫을 다 커버한다"는 테제는 시장 검증이 안 된 리스크다.**

- 각 버티컬엔 이미 강한 앱이 있다 (Flo·Premom, CareZone·Medisafe, 11pets·Dogo).
- 사용자가 정말 에셋을 전환하며 한 앱을 쓰는지, 아니면 *버티컬별 전문성*을 우선하는지는 **진짜 사용자 리서치가 필요**하다.
- 대안 포지셔닝 두 가지:
  - **A. "가족 케어 OS"** — 통합 유지, 타겟은 *"샌드위치 세대 보호자"* (본인 IVF + 어머니 항암 동시). 강력한 내러티브, 좁은 TAM.
  - **B. 싱글-버티컬 우선 출시** — 난임 한 축만 먼저 깊게, 이후 확장. 리스크 낮고 PMF 검증 빠름.
- 개인적 권고: **B 시작 → A 확장**. 현재 프로토타입의 세 에셋은 *내부 비전 문서*로서는 완벽하지만, v1 출시는 하나의 에셋 타입만 열고 나머지는 "곧 만나요" 대기열로 처리하는 편이 낫다.

## 5. 모바일 출시 로드맵 (제안)

### Phase 0 — 기반 (4–6주)
- Expo iOS/Android 빌드 파이프라인 (EAS Build).
- 백엔드 선택 (Supabase / Firebase / 자체 Nest).
- 인증 + 디바이스 동기화.
- 디자인 토큰화 + 다크 모드.
- 분석·크래시 리포팅(Sentry, Amplitude).

### Phase 1 — IA 완성 (6–8주)
- 에셋 단위로 4개 탭 모두 실장.
- 온보딩·에셋 생성 플로우.
- 빈/에러/로딩 상태.
- 설정·프로필·프라이버시 대시보드.

### Phase 2 — 플랫폼 네이티브 (6–8주)
- 푸시 + 로컬 리마인더 (조건부, 스누즈).
- iOS 위젯 + Live Activity / Android Glance.
- 생체 인증 잠금.
- 캘린더/헬스킷/헬스커넥트 연동.
- 오프라인 우선 동기화.

### Phase 3 — 공유·파트너 (4주)
- 초대 플로우, 권한 행렬.
- 실시간 싱크.
- 통합 알림 인박스.

### Phase 4 — 출시 준비 (4주)
- Privacy Nutrition Labels, Data Safety.
- 국내 PIPA/의료기기 SW 법리 검토.
- 앱스토어 에셋 (5초 후킹 영상, 에셋 타입별 스크린샷).
- TestFlight / Play Console 내부 테스트.
- KR 의료 인플루언서/커뮤니티 베타.

### Phase 5 — 출시 이후
- Apple Watch / Wear OS 동반앱.

## 6. 출시 전 반드시 결정해야 할 6가지 (의사결정 블로커)

1. **단일 버티컬 출시 vs 다중 에셋 출시** — 앞서 다룸. 6주 안에 결정 필요.
2. **수익 모델** — 프리미엄 per 에셋? 파트너 기능 유료? 프리미엄 인사이트? 가격 포지션(월 4,900원 vs 9,900원 vs 14,900원).
3. **타겟 국가** — KR 선공 후 확장? US/EU는 컴플라이언스 추가 작업 4–8주.
4. **파트너 동기화 모델** — 초대받은 사람도 계정 필수 vs 게스트 링크. 전자는 보안·후자는 전환률.
5. **의료 디스클레이머 톤** — "이건 의료 기기가 아닙니다"를 얼마나 전면에? 너무 강하면 신뢰 떨어지고, 너무 약하면 규제 리스크.
6. **데이터 삭제 정책** — 가입 해지 시 데이터 즉시 파기 vs 30일 유예 vs 익명화 보존. 법적·UX 양면에서 결정 필요.

---

## 7. 제가 CPO라면 당장 할 3가지

1. **사용자 리서치 스프린트 2주** — "에셋 여러 개를 진짜로 왕복하는 사람"이 실제로 존재하고 그게 주요 ICP인지 확증. 확증 못 하면 단일 버티컬로 스코프 축소.
2. **디자인 시스템 + 토큰 + 컴포넌트 라이브러리** 전담 인원 즉시 투입. 지금 미루면 Phase 2에서 피로 터짐.
3. **프라이버시를 마케팅 자산화** — *"당신의 IVF 데이터는 당신의 것입니다"* 내러티브는 경쟁 우위. 랜딩·스토어 설명·앱 내 모두에 관통시킬 것.

## 부록: 빠르게 확인한 기술·디자인 지표

- 스택: Expo + React Navigation. 모바일 iOS/Android 빌드 네이티브 래핑 이미 가능.
- 메타: `viewport-fit=cover`, `user-scalable=no` 설정 — 모바일 폼팩터 프라이머리 의도 확인.
- 번들 크기: `~54KB` HTML + 단일 JS 번들 → 네이티브 빌드 시 최적화 여지 큼.
- 접근성 태깅: 기본적인 탭/버튼 role은 있으나 스크린리더 레이블·라이브 리전 설정 불충분.
- 빈 탭 플레이스홀더 메시지 일관됨 ("이 탭은 다음 단계에서 채워질 예정이에요") — 개발 진행 관리 의도 드러남, 외부 빌드에서는 반드시 제거.
