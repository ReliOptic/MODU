## Problem Statement

TPO Funnel Runtime은 "app-native context orchestration" 개념을 제시하는 14개 스펙
문서 집합(Phase 0 완료)이나, 기존 personalization · onboarding · feature-flag 시장과
비교 시 "특별해 보이지 않는다"는 자체 진단이 있음. 본 세션은 이 진단을 해소할 OSS·
연구용 프레임워크로서의 명확한 wedge를 설계하고, Phase 1(첫 구현) 진입 경로를
확정하는 것이 목적.

## What Makes This Cool

TPO의 고유 "whoa" 장면은 **5줄의 structured context 입력이, 사람이 읽을 수 있는
decision trace로 즉시 변환되는 순간**이며, 이 trace가 기존 도구로 재현 불가한
이유는 네 블록의 동시 표기에 있음.

1. **Locked slots (business-fixed)**: 사업 모델상 고정되어야 하는 광고·프로모션·
   필수 상용 slot을 명시. `lockedSlots` 파라미터로 보호.
2. **Suppressed (with why-not)**: 왜 특정 컴포넌트가 배제되었는지의 역방향 이유.
   OpenFeature · LaunchDarkly · XState 모두 구조적으로 출력하지 않음.
3. **Abstention · Fallback**: 저신뢰 · 정책 거부 · 결측 컨텍스트에 대한 계층적
   안전 기본값 회귀 신호.
4. **Risk tier 경계 표시**: 위험 구간 초과 시 런타임이 감쇠한 결정을 명시.

이 네 블록의 결합은 기존 feature flag · 상태기계 · 마케팅 자동화 계열이 구조적으로
제공하지 않는 "app-native next-step 재배열의 설명 가능성"을 구성함.

## Constraints

- 일차 수신자는 엔드유저가 아니라 개발자. 평가 지표는 GitHub star · 채택 적용 시간.
- 본인 학습 기록 `quality-over-timeline` (2026-04-05, confidence 10/10): 시간제약
  기반 품질 저하 금지. 범위 축소는 허용, 품질 축소는 금지.
- 예제 · 문서의 도메인 색채는 의료 · clinic · procedure 계열을 제거. 이벤트 외생성은
  유지하되 정책 부담이 낮은 중립 도메인(conference · product launch · course enrollment)
  으로 교체.
- Ad-supported 앱과의 공존은 1차 primitive. `lockedSlots`를 첫 릴리스부터 지원.

## Premises (모두 승인됨)

1. **목적**: OSS · 연구용 프레임워크. 일차 수신자는 개발자. 상용 SaaS 유료화 설계는
   Phase 3 이후 유보.
2. **Wedge**: README 최상단 단일 예제는 "5줄 context → trace 출력"이며, 차별성의
   중심은 Suppressed · Fallback · Abstention · Risk tier. Selected 블록은 보조이며
   시각적 비중이 낮아야 함.
3. **도메인 중립**: README 예제는 conference · product-launch · course-enrollment
   중 하나로 고정. 의료 예시는 별도 "vertical pack" 섹션에만 등장.
4. **시드 범위**: Phase 0(문서) 다음은 전체 Phase 1 구현이 아니라 5줄 context +
   5줄 rule → trace 출력만 되는 최소 seed 라이브러리(~150–250 LoC). npm 패키지명·
   GitHub repo명·라이선스를 이 단계에서 확정.
5. **Locked-slot preservation**: TPO slot orchestration은 "모든 slot 재배치"가 아니라
   "사업 고정 slot은 locked, 주변 slot은 rearrangeable"의 이원 모델을 1차 primitive로
   포함. Trace 출력은 locked/flexible 구분을 명시.

## Cross-Model Perspective

본 세션에서는 Phase 3.5(Codex 독립 독해)를 실행하지 않음. 사유는 사용자 본인이
세션 내내 competitor analysis(OpenFeature · XState · LaunchDarkly · Braze) ·
self-bias diagnosis · 반례 제시를 반복적으로 수행하여, 독립 독해 수준의 교차 검증이
내부적으로 이미 이루어졌기 때문. Phase 1 착수 후 실제 사용자 피드백 수집 단계에서
외부 독해를 삽입하는 것이 가치 전달 측면에서 더 효율적.

## Approaches Considered

### Approach A: Minimal Seed (주말 완성)
- 단일 파일 TypeScript, ~150–250 LoC. 타입 정확성 · 테스트 커버리지 · trace 포맷
  세련도는 full quality.
- Export: `evaluate(context, rulePack, registry, lockedSlots) => TPOResult`.
- Result 구조: `lockedSlots · selected · suppressed(with why-not) · fallback ·
  abstention · riskTier · trace`.
- npm 패키지 `@tpo/seed` 또는 유사 이름. README는 단일 conference 예제.
- Effort: S. Risk: Low.
- Pros: 주말에 첫 Dev Whoa 가능. Premise 검증이 채택자 피드백 단위로 가능.
  스코프 크립 차단. 본인 quality-over-timeline 원칙과 정합.
- Cons: Policy gate DSL 성숙도 낮음. "그냥 rule engine"으로 축소 인식될 위험
  (README 문구로 방어 필요).

### Approach B: Full Architecture (전체 스펙)
- 9개 블록(02_RUNTIME_ARCHITECTURE.md) 전부 구현. 다중 패키지 분할.
- Effort: XL. Risk: Med (scope creep).
- Pros: 엔터프라이즈 신뢰감, 스펙 완성.
- Cons: Documentation-first failure mode 위험. 수 개월간 premise 미검증 상태 누적.
  첫 whoa 지연.

### Approach C: Trace Visualizer First
- 라이브러리 전에 브라우저 기반 TPO Trace 시각화 도구를 1차 배포.
- Effort: M. Risk: Med.
- Pros: Viral 가능성. 설치 없이 체험.
- Cons: 포지셔닝 혼선 가능.

## Recommended Approach

**A → B 단계 확장.** Minimal Seed를 Phase 1a로 주말 내 full quality 배포,
실제 채택자 피드백 1–2건 수집 후 그 피드백이 지시하는 블록부터 Full Architecture로
Phase 1b 증설. Approach C는 Phase 1a의 docs asset으로 후속 흡수 가능하므로
상호 배제가 아님.

## Success Criteria

- Phase 1a 완료 기준: 5줄 context + 5줄 rule을 입력하여, 네 블록(locked · selected ·
  suppressed · abstention)이 모두 포함된 사람 가독 trace를 출력하는 예제가 README에
  존재하고, npm install 후 30초 내 재현 가능.
- Premise 2 검증 기준: Phase 1a 공개 후 첫 1–2명의 채택자가 "trace가 팀 설득에
  쓰였다" 또는 "suppressed/why-not이 이 라이브러리의 이유다"에 해당하는 피드백
  제공.
- Premise 5 검증 기준: 첫 3개월 내 ad-supported 앱 또는 고정 슬롯 비즈니스 모델
  앱 개발자로부터 `lockedSlots` 사용 사례 보고 1건 이상.

## What I noticed about how you think

- **자기 바이어스 식별 능력**: 본인이 Phase 2B 두 번째 질문에서 "TPO 문서가 시간축
  바이어스 · 안전/정책 바이어스 · 여정 중심 바이어스 세 가지를 중첩시켰기 때문에
  fertility가 가장 잘 맞는 사례처럼 보인다"고 스스로 분해한 것은 창업자급 메타인지.
  대부분은 자신이 제시한 도메인을 변호하지 분해하지 않음.
- **업계 가격 구조에 대한 본능적 경고**: "1인 개발자는 차라리 개발하지, 레거시팀은
  그 정도면 돈 쓰자"라는 예측은 OSS 채택 곡선의 진짜 실패 지점을 정확히 지적.
  장밋빛 답이 아니라 이 수준의 솔직한 진단을 먼저 내놓는 것이 세션 품질의 출발점.
- **Load-bearing 제약을 뒤늦게라도 발견**: "광고 슬롯을 존중하면서 주변 UX를 재배열"이라는
  Premise 5 추가는 이 세션에서 가장 판세를 바꾸는 기여. 대부분은 "TPO가 광고와
  어떻게 공존하는가"라는 질문 자체를 런타임 설계가 끝난 뒤에 뒤늦게 접함. 본인은
  세션 중간에 이를 식별하고 문장까지 직접 제안.
- **정밀한 재프레이밍 습관**: "fertility는 설명이 잘 되지만 wedge 최적해는 아니다"
  같은 분리는 일반적이지 않음. "설명 친화성 ≠ wedge 적합성"이라는 구분을 스스로
  수행함으로써 진단 대화가 한 층 올라감.
