# Memo III — OSS 거버넌스 플레이북

## 1. 근본 질문부터

"오픈소스로 간다"는 **여섯 개의 분리된 결정**의 축약어입니다. 하나씩 풀어야 합니다:

1. **무엇을 오픈하는가** — 코어 라이브러리? 앱 전체? 백엔드? 혜택 KB?
2. **어떤 라이선스로** — MIT / Apache-2.0 / AGPL / SSPL / BUSL / Dual
3. **누가 저작권자인가** — 개인 기여자? 재단? 회사?
4. **누가 의사결정하는가** — BDFL? 메리토크라시? 재단 이사회? 투표?
5. **누가 운영비를 내는가** — 재단 기부? 회사 수익? 정부 지원?
6. **포크·캡처를 어떻게 막는가** — 라이선스? 상표? 커뮤니티? 재단?

대부분의 OSS 실패는 1~5 중 하나를 안 풀고 시작한 결과입니다. 이 메모는 당신의 맥락에 맞는 기본값을 제시합니다.

## 2. 무엇을 오픈할지 — 3층 모델

| 층 | 내용 | 공개 | 이유 |
|---|---|---|---|
| **Public Core** | 에셋·타임라인·리마인더·로컬 스토리지·암호화·동기화 프로토콜 | OSS 공개 | 신뢰·감사가능성. 경쟁자가 가져도 운영 못 함 |
| **Benefits Commons** | 국가별 지원·보험 KB, 자격 룰 | Creative Commons (CC-BY-SA) | 콘텐츠 커먼즈. 기여·검증 네트워크 유도 |
| **Operating Service** | 호스팅 서비스, 프리미엄 기능, B2B API, AI 인사이트 | 폐쇄 (상업 법인) | 수익 창출 + OSS 재투자 |

핵심 원칙: **인프라는 공개, 운영은 폐쇄, 지식은 커먼즈.**

## 3. 라이선스 선택 — 네 개의 현실적 후보

### 3.1 후보별 성격

- **MIT / Apache-2.0 (Permissive)**
  - 장: 채택 마찰 최저, 기업 기여 용이, 생태계 확대 빠름.
  - 단: **AWS·대기업이 재판매할 때 방어 수단 없음.** Elastic·MongoDB가 당한 시나리오.
  - 적합: 프레임워크·라이브러리 레이어(예: CareKit 수준).

- **AGPL-3.0 (Strong Copyleft)**
  - 장: 네트워크 서비스로 제공할 때도 소스 공개 의무. 클라우드 캡처 방지.
  - 단: 기업 사용자 일부 기피 (법무팀 블록).
  - 적합: 앱 본체·서비스 레이어. **이 제품의 Public Core에 1순위 추천.**

- **SSPL (Server Side Public License)**
  - 장: AGPL보다 더 강하게 관리형 SaaS 재판매 차단. MongoDB가 개발.
  - 단: OSI 비공인 라이선스. OSS 커뮤니티 일부 반감.
  - 적합: 순수 OSS 명분을 포기하더라도 캡처 방지가 최우선일 때.

- **BUSL (Business Source License)**
  - 장: 4년 뒤 Apache로 전환하는 지연 오픈. HashiCorp·Sentry 사용.
  - 단: 진정한 OSS 아님(공식적으로). 커뮤니티 신뢰에 부담.
  - 적합: 상업 레이어만 별도로 지연 공개.

### 3.2 권장 라이선스 매트릭스

| 컴포넌트 | 라이선스 | 비고 |
|---|---|---|
| Core Framework (에셋·타임라인 SDK) | **Apache-2.0** | 채택 극대화, 병원·정부 포함 |
| App Client (모바일 앱 본체) | **AGPL-3.0** | 포크 대응, 상표로 브랜드 보호 |
| Benefits KB | **CC-BY-SA 4.0** | 공유·기여형, 상업 이용 허용 |
| Premium Services Code | **폐쇄 / BUSL** | 상업 법인 자산 |
| Design System (토큰·컴포넌트) | **MIT** | 채택 마찰 최소 |

**핵심: 동일 코드베이스 내에서도 디렉터리별로 다른 라이선스가 가능합니다.** Elastic·Grafana가 모두 이 전략 사용.

## 4. 저작권 / 기여자 계약 (CLA vs DCO)

OSS 법인화에서 가장 많이 틀리는 지점입니다.

- **DCO (Developer Certificate of Origin)** — Linux 모델. 기여자가 커밋에 sign-off만. 저작권은 기여자에게 잔존.
  - 장: 진입 마찰 낮음. 실리콘밸리 선호.
  - 단: 라이선스 변경·상업화 유연성 낮음. 기여자 전원 동의 필요.

- **CLA (Contributor License Agreement)** — Apache·Google 모델. 기여자가 재단/회사에 라이선스 양도 또는 광범위 사용권 부여.
  - 장: 장래 라이선스 변경·dual license 가능.
  - 단: 기여자 마찰 ↑. 악용 시 신뢰 훼손 (Elasticsearch의 AGPL→SSPL 전환 때 논란).

**권장**: **DCO + 개인 기여자 / CLA(경량) + 기업 기여자** 이중 운영. Apache Software Foundation의 Individual CLA 모델을 참고. 법인은 저작권 자체는 가지지 않되 "광범위 영구 라이선스"를 보유하는 구조가 법적 안정성과 커뮤니티 신뢰 모두 균형.

## 5. 거버넌스 모델 — 네 가지 옵션

| 모델 | 예시 | 장점 | 단점 | 이 제품 적합도 |
|---|---|---|---|---|
| **BDFL** (Benevolent Dictator) | Python(Guido), Linux(Linus) | 빠른 결정, 비전 일관성 | 승계 위기, 번아웃 | 초기 적합 |
| **Meritocratic** | Apache | 투명·공정 | 느림, 대기업 유리 | 성숙기 적합 |
| **Foundation Board** | Mozilla, Linux Fdn | 기관 신뢰 | 정치화·관료화 | 장기 적합 |
| **Cooperative** | Drupal Association | 미션 일관성 | 자금 조달 난이 | 니치 |

**권장 진화 경로**: BDFL (창업자) → Core Maintainer Team (2년차) → Foundation Board (3~5년차). 시점별로 *자연스럽게 결정권을 분산*하는 서사가 커뮤니티 신뢰 구축의 핵심.

## 6. 재단 설립 — 국내외 선택지

### 6.1 한국에서

- **재단법인** (민법 제32조) — 공익 목적, 기본재산 요건(통상 5억 이상 권고, 주무관청별 상이). 주무관청: 과기정통부/보건복지부.
- **사단법인** — 회원 조직, 기본재산 요건 낮음. 이사 5인 이상.
- **지정기부금단체** 지정 (기획재정부) — 기부금 세제 혜택의 열쇠.
- **사회적기업 인증** (고용노동부) — 사회적 가치 지표 + 세제 혜택.
- **비영리민간단체** (행정안전부 등록) — 가장 낮은 허들, 세제 혜택 제한적.

**권장**: 단기 → 비영리민간단체 또는 사단법인으로 가볍게 시작 / 중기 → 재단법인 + 지정기부금단체 지정 병행.

### 6.2 해외 법인

- **Delaware 501(c)(3)** — 미국 기부 접근성. 국내에서 설립 복잡, 이중과세 이슈.
- **싱가포르 Charity** — 아시아 허브, 기부 파이 큼. 한국 국적 이사도 가능.
- **Swiss Association (Verein)** — Linux Foundation 모델. 유럽 표준, 글로벌 재단 다수 이용.

**권장**: 초기 한국 법인. 3년 차 이후 글로벌 자금·파트너십 확장 시 Swiss Verein 또는 Delaware 501(c)(3) 병행 법인 설립.

## 7. 이중 법인 구조 (Dual-Entity) — 검증된 패턴

가장 실전 검증된 OSS 미션 기업 구조:

### 7.1 Signal 모델
- **Signal Foundation** (501(c)(3)) — 미션 보유, 기부 수령, 핵심 자산(암호·프로토콜) 보유.
- **Signal Messenger LLC** — 운영 법인, 재단 100% 소유, 수익 없이도 재단 후원으로 운영.

### 7.2 Mozilla 모델
- **Mozilla Foundation** (501(c)(3)) — 미션·정책 옹호·커먼즈.
- **Mozilla Corporation** — 과세 법인, Firefox 상업화, Google 검색 계약으로 수익 → 재단 지원.

### 7.3 이 제품에 적용하면

```
┌────────────────────────────────┐
│   [가칭] 돌봄커먼즈 재단         │  ← 비영리 (한국 재단법인 + 지정기부금)
│   - Public Core 저작권          │
│   - Benefits Commons 운영       │
│   - 상표권 보유                 │
│   - 기부·공여 수령              │
└─────────┬──────────────────────┘
          │ 100% 소유 or 라이선싱
┌─────────▼──────────────────────┐
│   [가칭] 돌봄 PBC               │  ← 영리 (사회적기업 인증 주식회사 / B Corp)
│   - 앱 운영, 호스팅             │
│   - 프리미엄 기능 수익          │
│   - VC 투자 수령 가능           │
│   - 재단에 로열티·기부 환류     │
└─────────────────────────────────┘
```

**핵심 합의사항 (설립 시 문서화 필수)**:
- 재단이 상표권 보유 → 포크가 "돌봄커먼즈" 이름을 못 씀.
- 영리법인은 재단에 최소 X% 수익 기여 (Mozilla는 수익의 상당 비중을 재단 기여).
- 재단 이사의 과반수는 영리법인 이사와 분리 (이해충돌 방지).
- 영리법인 청산 시 잔여재산은 재단에 귀속.

## 8. 캡처·포크 방어 전략

OSS 역사에서 가장 큰 실패 패턴: AWS·대기업이 코드를 가져가 관리형 서비스로 재판매, 원 제작자는 수익 없이 유지비만 부담.

**이 제품의 방어 7선**:

1. **AGPL 또는 SSPL** — 관리형 재판매 차단.
2. **상표권** — "Dolbom Commons" 같은 이름·로고를 재단이 강하게 등록. 포크는 이름을 못 쓰게.
3. **Trusted Committer 모델** — 코어에 커밋 권한 소수·심사 기반.
4. **Contributor License Agreement** — 기여자가 재단에 광범위 라이선스 부여, 장래 방어 유연성.
5. **크로스-사일로 UX의 비-알고리즘성** — 크로스-에셋 오케스트레이션은 코드보다 UX·콘텐츠·운영 노하우 의존. 쉽게 복제 안 됨.
6. **Benefits Commons의 국가별 로컬 파트너십** — 자체 큐레이션 + 공식 기관 제휴는 포크가 따라올 수 없음.
7. **브랜드 신뢰의 선점** — "가장 오래되고, 가장 감사받은, 가장 많이 신뢰받는" 포지션 선점.

## 9. 커뮤니티 운영 — 실패하는 방식과 성공 방식

OSS 프로젝트의 73%는 **기술 문제가 아니라 커뮤니티 문제**로 실패합니다. 준비 체크리스트:

- [ ] Code of Conduct (Contributor Covenant 2.1 표준 채택)
- [ ] 기여자 온보딩 가이드 (CONTRIBUTING.md, 30분 내 첫 PR 가능 수준)
- [ ] Good First Issue 태그 30개 이상 사전 준비
- [ ] Triage 프로세스 문서 (이슈 대기열 48시간 내 1차 응답)
- [ ] Maintainer 로테이션 정책 (번아웃 방지)
- [ ] RFC 프로세스 (주요 변경의 공개 토의)
- [ ] Discord / Matrix 커뮤니티 채널
- [ ] 분기별 커뮤니티 콜
- [ ] 연간 기여자 리포트
- [ ] 커뮤니티 매니저 1인 채용 (프로젝트 규모 1,000 stars 넘으면 필수)

## 10. 한국 특수 이슈

- **공공 SW 조달에 유리**: 정부·공공기관 조달 시 OSS 가산점. 국내 SW 구매 예산의 기회.
- **과기정통부 OSS 지원 사업**: NIPA 공개SW 역량프라자, 개발자 대회, 정부 지원금.
- **개인정보보호법**: OSS라고 면책되지 않음. 서비스 법인이 전부 책임.
- **의료·보건 데이터**: 의료법·개인정보보호법 상 민감정보. 서비스 법인이 개인정보처리자. 재단은 KB·코드만.
- **세제 혜택**:
  - 지정기부금단체 지정 시 기부자 15~30% 세액공제.
  - 사회적기업 인증 시 법인세 50% 감면 (3년, 이후 50% 2년).
- **재단 설립 실무 난이도**: 주무관청 승인 3~6개월. 변호사·회계사 비용 1,500~3,000만원. 기본재산 5억 이상 권고(주무관청별).

## 11. 6개월 OSS 런치 로드맵

**M1**: 라이선스 결정 확정, CLA 템플릿, Code of Conduct, 이사회 초안 (3인 이상).
**M2**: 저장소 구조, README, CONTRIBUTING, 상표 출원 (KR·미국·EU).
**M3**: Core Framework 리팩토링 (의존성 정리, API 안정화), 문서 사이트.
**M4**: 내부 알파 공개 (초청 20명), Good First Issue 시드, Discord 열기.
**M5**: 공개 베타 (소프트 런치), 초기 기여자 온보딩, 분기 콜.
**M6**: 공식 런치 + 재단 설립 신청 접수 + 언론·커뮤니티 아웃리치.

## 12. III 결론

- **OSS는 가능하고, 이 카테고리에선 경쟁 우위.**
- **하지만 OSS는 "코드를 깃허브에 올림"이 아니라 "10년짜리 거버넌스 제도 설계"임을 인지해야 함.**
- **3층 공개 모델 + 이중 법인 구조 + AGPL 중심 라이선스 + Contributor Covenant CoC**가 이 맥락의 기본값.
- **초기엔 BDFL, 3년차에 재단 이사회 분리**가 현실적 경로.
- **상표·CLA·라이선스의 3중 캡처 방어**를 첫날부터 설계.

# Memo II — 10년 해자 설계: Semantic Memory Layer

## 1. 전제 — 무엇이 상품화되고, 무엇이 남는가

2026년 기준 이 앱의 기능 중 **5년 내 OS 기본 기능으로 흡수될 것들**:

| 기능 | 흡수 주체 | 시점 |
|---|---|---|
| 복약 리마인더 | iOS Health, Google Fit, Samsung Health | 이미 진행 중 (iOS 16+) |
| D-day 카운트다운 위젯 | OS 시계·캘린더 | 1~2년 |
| 주사 타임라인 | HealthKit + 약국 API | 2~3년 |
| 감정 체크인 | iOS 건강 앱 Mood (이미 있음) | 이미 진행 중 |
| 일정 관리 | OS 캘린더 + AI 보조 | 이미 진행 중 |
| 파트너 동기화 (일정·복약) | iCloud Family, Google Family | 2~3년 |

즉 **현재 홈 탭의 거의 모든 요소는 5년 내 상품화**됩니다. 해자가 아닙니다.

반면 **10년이 지나도 플랫폼이 흡수할 수 없는 레이어**가 존재합니다. 이것이 해자의 후보입니다.

## 2. 10년 남는 레이어 — "Semantic Memory" 5종

### 2.1 WHY 레이어 (결정의 이유)

플랫폼은 *"언제 어떤 약을 복용했는가"*를 기록합니다. **왜 이 약을 선택했는지, 왜 이 병원을 옮겼는지, 왜 이 주기에 치료를 중단했는지**는 아무도 기록하지 않습니다.

- 의료 결정의 맥락은 본인조차 6개월 뒤 잊음.
- 의사가 바뀌면 사라지는 맥락.
- 차세대 의료 AI가 가장 필요한 입력 데이터.

**UX 원형**: 에셋의 상태 변경 시점마다 *"왜 지금 이 결정을 했어요?"* 프롬프트. 음성 30초 노트도 OK. 타이밍은 변화 직후 1~24시간 내.

### 2.2 PATTERN 레이어 (자기이해)

사용자가 자기 몸·자기 루틴을 읽어가는 언어. *"밤 10시 이후 먹으면 다음 날 메스꺼움이 심하다"*, *"배란일 전 3일간 유독 예민하다"*, *"어머니는 오후 컨디션이 좋다"*.

- 플랫폼은 *집계 인사이트*를 제공하지만 *개인 언어로 된 자기이해*는 제공하지 않음.
- 이게 10년 쌓이면 사용자 본인의 건강 자서전.

**UX 원형**: 주간 리뷰에서 *"이번 주 몸이 가장 좋았던 순간은?"* → 자유 텍스트. 연말에 AI가 요약해 *"올해의 나 패턴"* 생성.

### 2.3 HANDOFF 레이어 (돌봄의 계승)

어머니가 돌아가신 후 어머니의 케어 기록이 어디로 가는가? 당신이 돌아가셨을 때 자녀가 당신의 의료 여정을 이해하는 방법은?

- 아무도 풀지 않는 문제. 플랫폼은 사용자 사망 시 계정 동결.
- 돌봄 유산(care legacy)은 세대 간 가치 전달의 미개척지.

**UX 원형**:
- 에셋별 "유언 모드" — 사전에 공동 돌봄자·상속자 지정.
- 사망 시 자동 전환: *"어머니 항암"* → *"어머니 기억"* (회고용 아카이브).
- 다음 세대가 *"어머니 항암 때 우리가 한 결정들"*을 읽을 수 있는 읽기 전용 뷰.

이건 진짜 독보적입니다. 의료 플랫폼이 절대로 가지 않을 영역입니다.

### 2.4 NARRATIVE 레이어 (이야기)

단순 로그가 아니라 사용자가 직접 쓰는 여정 서사. 자기공명 일기.

- 시험관 3회차 실패 후 "이번엔 왜 안 됐을까" 노트.
- 어머니 5차 항암 후 "오늘은 어머니가 웃었다" 기록.
- 보리 검진 후 "얼굴 윤곽이 약해 보였다" 메모.

**UX 원형**: 시점별 *"오늘의 한 줄"* 권장. 월말에 전체 한 줄들이 타임라인으로 재생됨.

### 2.5 COMMUNITY SEMANTIC 레이어 (집합 의미)

*동의 기반으로* 개인의 의미 기록이 커뮤니티 지식으로 환원. *"시험관 3회차 실패 후 가장 도움이 된 것은?"*에 대한 실제 사용자 언어의 집합.

- Reddit·맘카페·환우회 커뮤니티가 쌓아온 자산을 구조화된 형태로.
- 검색·임베딩으로 *"나와 가장 비슷한 여정을 걸어간 사람들의 선택"* 탐색.
- **집합 학습이 아니라 집합 의미** — 모델을 학습시키지 않고도 RAG로 제공.

## 3. 데이터 아키텍처 — 10년 썩지 않게

### 3.1 이벤트 소싱 + 이뮤터블 append-only

- 모든 기록은 이벤트. *"배란일 변경"* 같은 업데이트도 새 이벤트로.
- 원본 잔존 → 10년 뒤에도 맥락 복원 가능.
- 동의 철회 시 이벤트 *삭제*가 아니라 *마스킹* (법적 삭제 권리는 별도 계층에서 처리).

### 3.2 Semantic Markup

- 모든 이벤트에 *왜*, *기분*, *관련 결정* 필드가 옵션으로 있음.
- AI가 Markup을 자동 제안(*"이 변화는 이전 회차 약 변경과 관련 있어 보여요. 연결할까요?"*).

### 3.3 사용자 소유 + 데이터 주권

- 로컬 우선 + E2EE 동기화.
- 언제든 NDJSON / FHIR R5 / Markdown 묶음으로 내보내기.
- 본인 소유 IPFS / Solid pod 연동 옵션 (장기 미래, 데이터 주권 내러티브).

### 3.4 FHIR 호환성

- FHIR R5 Observation / Condition / MedicationStatement 매핑.
- 병원 EMR 연동의 미래 가능성 확보.
- 한국 보건의료정보원 "우리병원 마이데이터" 프로토콜 대응.

## 4. LLM 시대의 해자 — Personal vs Platform AI

2030년에는 모든 사람이 개인 AI 비서를 가질 것입니다. 그 비서는 당신의 Apple Health, Google Fit, 이메일, 캘린더를 볼 것입니다. **그런데 의료 결정의 *맥락*은 아무도 안 갖고 있을 것입니다.**

이 앱의 전략적 포지션: **"당신의 개인 AI 비서가 참조하는 의료 여정의 근원 메모리 소스"**.

- 앱 안에 가둬진 AI가 아니라, 앱이 사용자의 개인 AI에게 *"내 10년치 돌봄 이야기"*를 공급.
- MCP(Model Context Protocol) 같은 표준 인터페이스 채택.
- 사용자의 비서 AI가 *"3년 전 어머니가 같은 증상이었을 때 어떻게 했지?"*에 답할 수 있게.

**이게 10년 해자입니다.** 지금 시작하면 2036년엔 *가장 오래된 구조화된 돌봄 서사 데이터셋*의 보유자.

## 5. 구체적 구현 기능 — 해자를 만드는 6개

1. **Decision Journal 패턴** — 에셋 상태 변경 시점마다 WHY 프롬프트.
2. **Weekly Retrospective** — 매주 일요일 *"이번 주 가장 도움된 것 / 가장 힘든 것"* 리추얼.
3. **Letter to Future Me** — 특정 시점(회차 종료, 이식 D-day)에 미래의 자신에게 편지. N개월 뒤 재생.
4. **Care Handoff Protocol** — 가족 간 케어 인계 시 "이 사람에 대해 알아야 할 것" 자동 요약.
5. **Life Event Markers** — 결혼·이사·실직 같은 생활 이벤트와 의료 데이터 크로스 레퍼런스.
6. **Annual Memoir Export** — 연말에 "올해의 돌봄 서사"를 e-book·PDF로 내보내기. 물리적 인쇄 가능.

이 6개는 지금의 플랫폼 경쟁자들이 *하지 않는* 것들입니다. 그리고 시간이 지날수록 가치가 붙습니다.

## 6. 플라이휠 재프레임

전통적 "데이터 → 모델 → UX → 유저" 플라이휠 대신:

**Semantic Memory Flywheel**:
```
더 깊이 기록 → 개인 서사의 질 ↑ → 스스로에게 유용 ↑
     ↑                                       ↓
   신뢰 ↑                                   리텐션 ↑
     ↑                                       ↓
공개 투명성/OSS                    오래 쓸수록 이탈 불가
```

이 플라이휠은:
- 집합 학습 없이 돎 (프라이버시 정합).
- 사용자 이탈 비용이 *기능*이 아니라 *서사*에 걸림.
- 10년 쌓인 서사는 대체재가 없음 (Figma → 다른 툴은 가능, 10년 일기 → 다른 곳으로 못 감).

## 7. 피해야 할 안티패턴

- **게이미피케이션 과잉** — 스트릭·배지·레벨은 돌봄 맥락과 충돌. *"5차 항암 완료! 🏆"*는 역겨운 UX.
- **소셜 피드** — 공개 타임라인 금지. 커뮤니티는 익명·읽기 우선·구조화된 Q&A로.
- **AI 자가진단 유혹** — *"이 증상은 X일 수 있어요"* 같은 진단 시사는 규제·신뢰 양면에서 파멸. 해석·질문 정리에 머물러야 함.
- **데이터 수익화** — 익명화라도 건강 데이터 매각은 브랜드 자살. 기부 기반 리서치 커먼즈 모델만 허용.
- **과도한 자동화** — 의미 기록은 수동이어야 의미가 남음. *"AI가 요약해드릴까요?"*는 원자료 보존 후 보조 기능.

## 8. 10년 지평의 단계

- **Year 0~2**: Core 로깅·리마인더 PMF, Semantic Memory 기능 시드 (WHY 프롬프트, 주간 회고).
- **Year 3~5**: Handoff·Narrative 기능 성숙, Benefits Commons 확장, 개인 AI 비서 연동 표준.
- **Year 6~10**: 세대 간 케어 유산 플랫폼으로 포지셔닝, 의료 AI 시대의 *"맥락 제공자"* 레이어, 연구 커먼즈 기여자 100만+.

## 9. II 결론

- **지금 빌드의 기능은 5년 내 OS에 흡수됨.** 해자 아님.
- **해자는 "의미의 기록" — WHY, PATTERN, HANDOFF, NARRATIVE, COMMUNITY 5종.**
- **이벤트 소싱 + 사용자 소유 + FHIR 호환 + MCP 공급**이 데이터 아키텍처 원칙.
- **"플랫폼이 접근 못 하는 레이어에서 10년 복리"** 가 CEO의 제품 북극성.
- **LLM 시대에 당신의 비서 AI가 참조하는 돌봄 서사의 소스**가 최종 포지션.

# Memo I — 법인 구조 의사결정 메모

## 1. 결정할 항목

- **주 법인 형태**: 주식회사(영리) vs 사회적기업 주식회사 vs 비영리(재단·사단) vs 협동조합 vs 하이브리드.
- **관할**: 한국 단일 vs 한국 + 해외 법인.
- **미션 락 기제**: 정관 조항 vs B Corp 인증 vs 사회적기업 인증 vs Perpetual Purpose Trust.
- **투자 유치 가능성**: VC vs 재단 공여 vs 매출 자기자본 vs 정부지원.
- **창업자 지분·권한 구조**.

## 2. 세 가지 현실적 선택지

### 옵션 A — 순수 영리 (일반 주식회사)

- **구조**: 주식회사. VC 투자 유치 쉬움. Exit 경로 (M&A, IPO) 표준.
- **장**: 속도. 자금 조달 용이. 인재 영입 유인 명확(스톡옵션).
- **단**: 미션 드리프트 리스크. 투자자 압력으로 데이터 수익화·광고 모델 탈선 가능. *신뢰* 중심 카테고리와 부적합.
- **이 제품 적합도**: 중하. PMF 검증 속도는 빠르나, 브랜드 신뢰의 장기 자본 손실.

### 옵션 B — 순수 비영리 (재단법인)

- **구조**: 비영리 재단법인. 기부 + 정부 공여 + 국제 공여 의존.
- **장**: 미션 일관성. 기부자 세제 혜택. 공공 조달 유리. 국제 공여 접근.
- **단**: 성장 자본 조달 느림. 인재 영입 유인 제한. 운영 유연성 부족. 주무관청 규제.
- **이 제품 적합도**: 중. 순수 커먼즈라면 적합하나 상업 서비스를 병행해야 함.

### 옵션 C — 하이브리드 (재단 + 영리 PBC) ★ 권장

- **구조**: 비영리 재단 (코어·상표·KB 보유) + 영리 주식회사 (운영·수익화).
- **장**: 미션 락 + 성장 자본 병행. Signal·Mozilla 검증 패턴. ESG/PRI 투자 수용. 국제적 정당성.
- **단**: 설립·운영 복잡성 ↑. 이사회 2중. 법무·회계 비용 ↑. 초기 $200~500K 수준의 구조 비용.
- **이 제품 적합도**: **최상.** 신뢰 카테고리 + 글로벌 지향 + OSS 병행에 가장 부합.

## 3. 권장 법인 구조 상세 설계

### 3.1 재단법인 (가칭 "돌봄커먼즈 재단")

- **형태**: 민법상 비영리 재단법인 + 과기정통부 또는 보건복지부 주무관청.
- **기본재산**: 5억원 (창업자 출연 또는 초기 기부자 풀). 3년 내 지정기부금단체 지정 신청.
- **이사회**: 최소 5인. 창업자 + 의료 전문가 + 기술 전문가 + 환자 옹호 대표 + 독립 이사 1인 이상.
- **자산**: Public Core 저작권, 상표권, Benefits Commons 콘텐츠, 브랜드.
- **수입원**: 기부, 정부지원, 국제 공여 (WHO·Gates Foundation·KOICA), 영리법인 로열티/기부 환류.
- **지출**: 코어 유지관리, 커뮤니티 운영, 정책 옹호, 장학·리서치 공여.

### 3.2 영리 주식회사 (가칭 "돌봄 주식회사")

- **형태**: 주식회사 + 사회적기업 인증 + B Corp 인증 (2~3년차 목표).
- **소유 구조**:
  - 재단 25~49% (지배구조적 영향력, 이사 지명권).
  - 창업자 30~40%.
  - ESOP 15%.
  - 투자자 10~30%.
- **정관 미션 락 조항**:
  - "본 회사는 돌봄 저니의 디지털 권익 확대를 주요 목적으로 한다."
  - 사업 전환·청산 시 재단 동의권.
  - 데이터 수익화·광고 모델 진출 시 재단 거부권.
- **수익 모델**: 프리미엄 구독, B2B 파트너십, 보험·고용주 연계.
- **수익 환류**: 세전 이익의 최소 10% 재단 기부 (또는 상표 사용료 형태).

### 3.3 주주간 계약의 필수 조항

- **Drag-along / Tag-along**: 표준 조항 + 미션 수호 예외.
- **Liquidation Preference**: 1x non-participating 권장 (욕심 부리지 말 것).
- **Anti-dilution**: Broad-based weighted average.
- **창업자 베스팅**: 4년 + 1년 절벽. 미션 위반 해임 시 부베스팅 가속 없음.
- **미션 수호 조항**: 기본 정관 변경은 75% 이상 + 재단 동의.

## 4. 조세 / 규제 분석 (한국 기준)

### 4.1 재단 측

- **지정기부금단체 지정** (기획재정부 고시): 기부자 개인 15~30% 세액공제, 법인 기부자 10~50% 손금산입.
- **상속·증여세 면제**: 공익 목적 자산 출연.
- **법인세**: 고유목적사업 수익은 비과세, 수익사업은 과세.
- **부가가치세**: 비영리라도 수익사업은 과세 대상.

### 4.2 영리 법인 측

- **사회적기업 인증 시**: 법인세 50% 감면 (3년 → 이후 50% 2년), 부가세 경감.
- **벤처기업 인증 병행 가능**: R&D 세액공제, 특허 감면.
- **스톡옵션 비과세 한도 확대** (2024년 개정, 연 2억원).
- **데이터 3법 대응**: 개인정보처리자로서 정보보호 체계 구축 의무.

### 4.3 의료·보건 특수 이슈

- **의료기기 소프트웨어(SaMD) 해당 여부**: 현재 기능(리마인더·기록)은 비해당 추정. 단, AI가 복약 변경을 시사하는 순간 의료기기 해당 가능성. **식약처 사전 자문 권장** (설립 6개월 내).
- **원격의료 규제**: 의사 상담 기능 도입 시 의료법 충돌. 지금은 지양.
- **보험업법**: 보험 추천 모델 시 전자금융업 또는 보험대리점 라이선스 필요 가능.

## 5. 해외 법인 — 3년 로드맵

- **Year 1**: 한국 재단 + 한국 영리.
- **Year 2**: PMF 확인 시 미국 Delaware C-Corp 자회사 설립 (글로벌 투자 유치 시 관례적).
- **Year 3**: Swiss Verein 또는 Delaware 501(c)(3) 자매 재단 설립 (국제 공여 수령용).

## 6. 창업자·초기 팀 관점

- **CEO 지분**: 일반 30~40% 시작 → VC 라운드 거치며 15~25% 수준 정착.
- **Common-Preferred 분리**: 초기 Common만. Series A 이후 Preferred 시리즈 도입.

## 8. 투자 유치 시 주의점 (구조 때문에)

- **Cap Table Explainer**: 이중 법인 구조를 VC가 이해하는 데 2~3회 미팅 추가 필요.

- **Exit 내러티브**: M&A 가능하되, 인수자가 미션 락 조항을 승계. "인수는 가능, 해체는 불가" 구조.

## 9. 의사결정 순서

1. **Week 1~2**: 창업 공동대표 간 미션·가치·10년 지평 합의. 서면화.

## 10. I 결론

- **하이브리드(재단 + 영리 PBC) 구조가 당신의 미션·카테고리·시간지평에 가장 부합.**
- **단기 비용은 크나, 10년 관점에서 미션 드리프트·신뢰 훼손 리스크의 보험료로 정당화.**
- **한국 시작 + 3년 내 해외 확장**이 현실적 시퀀스.
- **식약처·기재부·과기정통부 자문을 설립 6개월 내 모두 받을 것.**
- **CEO 본인이 "미션을 위해 10~15년 매일 수 있는가"를 먼저 자문.** 이 구조는 그 대답이 "예"일 때만 맞습니다.

# 통합 결론 — 세 메모가 한 방향을 가리키는 이유

세 메모는 같은 질문의 세 각도입니다:

- **III (OSS)**: 어떻게 *신뢰를 인프라화*할 것인가.
- **II (10년 해자)**: 어떻게 *의미를 복리화*할 것인가.
- **I (법인 구조)**: 어떻게 *미션을 제도화*할 것인가.

이 셋의 공통 답은 **"플랫폼이 되는 게 아니라 커먼즈가 되는 것"**입니다. Flo·CareZone·11pets는 플랫폼입니다. 이 제품의 기회는 *돌봄의 디지털 커먼즈*가 되는 것. 그 커먼즈 위에 당신의 영리법인이 *가장 잘 운영하는 사업자*로 서는 것.

10년 뒤 이 프로젝트의 성공 모습을 상상해보면:

- Wikipedia가 지식의 커먼즈이듯, *이 프로젝트는 돌봄 여정의 커먼즈*.
- Signal이 보안 통신의 *신뢰 표준*이듯, *이 프로젝트는 돌봄 데이터의 신뢰 표준*.
- 개인의 AI 비서가 당신의 10년치 돌봄 서사를 참조하는 *근원 소스*.

이게 가능하다면, 이건 제품이 아니라 **세대 간 인프라**입니다. 그리고 그것이 CEO 본인이 가장 원하는 답의 형태였을 것이라 생각합니다.
