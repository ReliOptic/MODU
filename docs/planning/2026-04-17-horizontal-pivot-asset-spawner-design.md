## Problem Statement

기존 mobile app 시장은 vertical-per-life-situation (난임 앱, 학습 앱, 운동 앱, 프로젝트 관리 앱) 으로 fragmented 되어있다. 사용자는 인생의 chapter 가 바뀔 때마다 새 앱을 찾고, 각 앱은 한 chapter 가 끝나면 가치를 잃는다 (난임 앱 → 임신 후 무용; 학습 앱 → 졸업 후 무용; 운동 앱 → 부상 시 무용).

MODU 는 이 단편화를 메타-platform 으로 해소한다. 한 앱이 사용자의 현재 life chapter (= **asset**) 에 맞춰 UX·위젯·메뉴·알림을 metamorphic 하게 재구성한다. chapter 가 바뀌면 새 asset 을 spawn 하고, 동시 진행 중인 multi-asset 도 지원한다 (난임 + 직장 프로젝트 + 골프 동시).

핵심 thesis: AI cost-collapse 가 metamorphic UX 의 dev cost 를 historically 처음으로 startup-affordable 하게 만들었다. 이전엔 사람 dev 가 vertical 마다 hand-build 해야 했지만, 이제는 LLM-driven asset configuration + adaptive UI presets + TPO-routing 으로 한 코드베이스가 무한히 많은 vertical 을 cover 한다.

## Demand Evidence (deferred per P3)

명시적 demand evidence 는 본 doc 작성 시점에 수집되지 않았다 (Q1 skip = 의식적 risk premise P3). 본 섹션은 placeholder 가 아니라 의도적 deferral — founder 가 ship-then-discover 경로를 선택했음을 명시. Founder 의 prior 근거: 본인의 IVF 경험 + KR/global mobile app fragmentation 패턴 관찰 + AI-era thesis (P2).

수집은 prototype-ship 후 시작 (The Assignment 참조). 14-30일 내 측정 항목:
- TestFlight beta release D7 retention ≥30%
- 2nd-asset-creation rate ≥15% (horizontality 의 단일 falsifiable 시그널)
- < falsifier threshold → vertical-template-marketplace 로 pivot 검토 (subagent 제안, founder 가 hold)

## Status Quo

현재 사용자가 living with 인 workaround 패턴:
- KakaoTalk + Google Calendar + Notion + 종이 다이어리 + 의사 상담일지 + 가족 공유 = 분산
- 각 vertical 앱 (난임 = 마미스토리/난임센터; 학습 = 클래스123/오르빗; 운동 = 스트라바/캐디톡) → 한 챕터 종료 후 탈퇴
- 시간이 지나면 chapter context 가 사라짐 (기억은 사람 머릿속, 데이터는 흩어짐)

이 workaround 의 cost: mental overhead, context loss, 가족·의료진 간 정보 동기화 실패. MODU 의 가설은 metamorphic single-app 이 이 cost 를 흡수한다는 것.

## Target User & Cold-Start Surface

**Cold-start primary**: "이번 달 인생에서 뭘 챙기고 있는 사람" — active life chapter 보유자. KR 첫 release.

**Warm-start vertical seed**: Fertility (IVF/IUI cycle) — 기존 66 파일 sunk cost 활용. 신규 사용자가 fertility 를 "예시 자산" 으로 보고, 본인 자산을 spawn 할 수 있게 1st-class 노출.

**Long-arc target**: Global. Multi-asset per user. Network effect via family/partner/care-team sync.

## Premises

- **P1 — Positioning**: MODU 는 v1 부터 horizontal metamorphic life-asset platform. ADR-0004 (fertility-only v1) 폐기. fertility 는 여러 asset 중 하나의 warm-start seed.
- **P2 — AI 시대 thesis**: 생성형 + 적응형 UI + LLM-driven 자산 구성이 metamorphic UX 를 historically 처음으로 startup-affordable 하게 만들었다. binding constraint = build velocity + meta-context routing 품질, NOT architecture risk.
- **P3 — Demand-before-build 거절**: 첫 100명을 인터뷰로 미리 찾지 않는다. ship → discover. Founder 가 의식적으로 받아들인 risk.
- **P4 — Long-term moat**: compound user-input + TPO-based context accumulation + metamorphic 적응 → 사용자당 가치가 시간에 대해 비선형 증가. B2G + 양면시장 + contribution margin 경제학의 실질 기반. Founder 가 cross-model challenge 를 reasoning-based 로 reject (AI-native 와 prior-era 의 epistemological 분리). 본 doc 에서는 P4 유지하되 instrumentation 으로 운영 지표화.
- **P5 — Operating order**: "regulation → data mechanism → UX" 순서 유지하되 per-asset compliance matrix 로 재구조화 — 학생 = 미성년자 보호 + E2; 직장 = 회사 IP/NDA risk + E1; 건강 = Art.9 + E4; 운동 = E1 광범위. asset 생성 시 envelope 자동 매핑.

## Cross-Model Perspective

독립 Claude 서브에이전트 (대화 맥락 미공개) cold-read 결과 — 핵심 인사이트 4건:

- **Steelman**: "the wedge is not a vertical — it is the engine itself." LLM cost-collapse 가 per-vertical configuration cost 를 무너뜨려 단일 codebase 가 무한 vertical 을 동시 서비스. 사용자당 append-only context store = 이탈 비용 비선형 상승. 집계 시 B2G 의 first dataset.
- **가장 revealing 한 founder verbatim**: *"결국이걸 얼마나 빨리 만들어낼거냐 그게 줄요한거에요."* → MODU 의 실제 제품은 app 이 아니라 internal asset-authoring pipeline. App 은 demo. 새 자산 ship 시간이 >1일이면 thesis 죽음, <1시간이면 unstoppable. 본 doc 의 instrumentation 항목으로 박힘.
- **제기된 P4 challenge** (founder reasoning 으로 reject): Notion/Evernote/Day One/Roam 모두 "compound personal context" 약속 → 40%+ 연간 churn. 사용자는 과거 query 하러 돌아오지 않음. ChapterMemory 의 compounding 은 MODU side (training signal) 에 대해서는 옳고 user side (retention moat) 에 대해서는 falsifiable 가설. Founder rebuttal: "AI-native 가 prior-era 분석을 무력화한다." — P4 유지, instrumentation 으로 measured.
- **권장 14일 지표**: D7 retention ≥30% AND 2nd-asset-creation ≥15%. 두 번째 자산 생성 = horizontality 의 단일 falsifiable 시그널.

## Approaches Considered

### Approach A: Asset-Spawner-First (chosen, high-fidelity)
첫 화면 = 대화형 onboarding. LLM 이 사용자 답변을 받아 TPO 스키마 + 위젯 + daily nudge 를 spawn. fertility 1개 자산은 baked seed (sunk cost 활용), 새 자산 spawning 은 1st-class 노출. 멀티에셋 from day 1. 음성·사진·텍스트 입력 모두 production 품질. local-first + sync-ready 아키텍처. 7-12일 sprint 후 TestFlight.

### Approach B: Multi-Asset Showcase + Spawner
4개 baked asset 모두 살리고 spawner 추가. 14-20일. polish 최대지만 confounded A/B (warm seed 의 polish 차이가 spawn 의 약점을 가림).

### Approach C: Pipeline-First (lateral)
사용자 미release. 내부 asset-authoring pipeline 먼저 검증 — 5개 신규 자산 1시간 이내 ship 가능 증명. 이후 release. P2 깨끗이 검증되나 시장 신호 늦음.

## Recommended Approach

**A (Asset-Spawner-First, high-fidelity).** 이유: P1 + P2 동시 in-market 검증, sunk cost 활용 효율, 사용자 contact 가장 빠름. C 의 internal pipeline 검증은 A 의 첫 자산 spawn N건으로 자연스럽게 같이 검증된다 (별도 pipeline-only sprint 불필요).

**고도화 원칙 (founder 지시):** 모든 기능은 production 품질로 ship. "skip X" / "v0.2 로 미룬다" 형태의 carve-out 금지. 모든 기능이 ship 되되, 단계별 sequencing 으로 land 한다.

**v1 ALL-IN Scope (이 모든 항목이 release 안에 들어옴, 내부 sequencing 만 다름)**:

1. **대화형 Onboarding (formation flow 위에서 확장)**
   - 음성 입력 (STT via Edge Function — Whisper)
   - 사진 첨부 (R2 presigned upload pipeline)
   - 텍스트 입력 + preset 선택지 동시 노출
   - LLM (Claude Edge Function) 이 응답 → asset schema 생성

2. **Asset Spawner (LLM-driven runtime asset generation)**
   - input → TPO 스키마 (T: 다음 일정, P: 지역 + 정부 정책, O: 상황 컨텍스트)
   - 위젯 3-5개 (preset 컴포넌트 라이브러리에서 LLM 이 조합 선택)
   - daily nudge 1개 (push notification 으로)
   - asset palette (5색 중 LLM 이 콘텐츠 톤 매칭)
   - **LLM contract (JSON schema, drafted)**:
     ```json
     {
       "asset_id": "uuid",
       "title_ko": "string",
       "category_hint": "fertility|study|work|fitness|petcare|caregiving|hobby|other",
       "envelope": "E1|E2|E3|E4",
       "palette_hint": "dawn|mist|blossom|sage|dusk",
       "tpo": {
         "time": { "anchor_event_kind": "string", "rrule": "string|null", "next_at": "ISO8601|null" },
         "place": { "needs_geo": "bool", "policy_query_terms": ["string"] },
         "occasion": { "context_summary_ko": "string", "tone": "ritual|tactical|playful|gentle" }
       },
       "widgets": [
         { "preset_id": "string", "props": { "<preset-specific>": "any" } }
       ],
       "nudge": {
         "trigger": "time|tpo-change|user-asked",
         "default_local_time": "HH:MM",
         "copy_ko": "string",
         "max_freq_per_day": 1
       }
     }
     ```
   - **Failure modes (named, not deferred)**: (a) LLM 이 envelope 을 잘못 매핑 → server-side validator 가 reject, fallback to E2 + warn; (b) 알 수 없는 preset_id → server 가 nearest match 로 substitute + log; (c) 위젯 5개 초과 → trim to 5; (d) STT 인식 실패 → text 입력 fallback.
   - **eval harness**: 30개 seed 시나리오 (각 vertical 5개) 로 spawn 결과의 envelope 정확도 + preset 매칭 합리성 + 사용자 만족 (1-5 thumbs) 측정. Sprint 1 deliverable.
   - **두 metric 분리**:
     - `asset_authoring_time_internal`: founder/팀이 새 preset 또는 새 envelope 정책 등록 → ship 까지 (<1시간 target, P2 의 internal 검증)
     - `asset_spawn_time_user`: 사용자 입력 시작 → 첫 위젯 렌더 (<2분 target, UX 검증)

3. **멀티에셋 from day 1 (asset switcher — bottom sheet native + dropdown web)**
   - 사용자가 동시에 N개 asset 보유 가능
   - 각 asset 별도의 ChapterMemory append-only 타임라인
   - quick switching ritual (920ms — fade-out 280 + ritual hold 360 + fade-in 280) per §6.1

4. **TPO Routing Engine (live)**
   - T: Calendar + 다음 일정 focus widget
   - P: Geolocation + 지역별 정부 혜택/규제 lookup
     - **Vector DB**: pgvector on Supabase (Seoul region) 또는 Cloudflare Vectorize
     - **Data source v1**: 정부24 (gov.kr) 공공 데이터 포털 의 복지·지원·정책 dataset (open API + bulk download), 보건복지부 / 교육부 / 문화체육관광부 / 고용노동부 의 RSS · 공식 페이지 크롤링 (robots.txt 준수)
     - **Ingestion plan**: Sprint 1-2 에서는 fertility (난임 시술비 지원) + study (학자금 + 학습 바우처) + childcare (어린이집 + 출산지원금) 3개 vertical 우선. asset 마다 ~50-200 정책 record. nightly cron (Edge Function scheduled) 으로 갱신
     - **Latency**: 사용자 query → top-k=5 retrieve <300ms (Edge cache 활용)
     - **Out of scope v1**: 외국 (US/EU) 정책 — Phase 9 deferred
   - O: 사용자 input + asset context 조합

5. **ChapterMemory (append-only, local-first, sync-ready)**
   - visibility tagging (self/partners/doctor) at write-time
   - semantic retrieval via embeddings (Gemma local 우선, Claude fallback)
   - one-tap export (.zip JSON+사진), one-tap delete (즉시 파기)

6. **Push Notification System (real, not in-app only)**
   - daily nudge per asset (사용자가 시간 선택)
   - TPO 기반 contextual push
   - 사용자 정의 절제 가능 (per-asset on/off, frequency cap)
   - lock-screen 위젯 (CPO review 항목)

7. **Privacy + Consent Layer (onboarding 에서 plain 하게)**
   - local-first explanation
   - opt-in sync (post-bonding moment trigger)
   - opt-in aggregate research (B2G hidden surface 의 plain 한 표면)
   - app lock default ON (생체인증)

8. **Per-asset Compliance Matrix (P5)**
   - asset 생성 시 envelope 자동 매핑 (LLM 이 1차 매핑, server validator 가 sanity-check)
   - **Envelope 정의 (본 doc 의 자체 정의 — Phase 1 event schema doc 의 E1-E4 와 정합)**:
     - **E1 — 일반 (default)**: 일반 lifestyle/productivity 데이터. consent: standard ToS. retention: user-controlled. examples: 직장 프로젝트, 운동, 자격증, 취미
     - **E2 — 미성년 / 교육 데이터**: COPPA 유사 보호 + KR 개인정보보호법 미성년 가이드. consent: parental for <14, simplified for ≥14. examples: 학생 학습, 입시, 청소년 활동
     - **E3 — 가족·관계 데이터**: 본인 외의 사람 데이터 포함 (배우자·자녀·부모 케어). consent: explicit "타인 정보 입력" 안내. shareable scope 명시. examples: 자녀 케어, 부모 부양, 결혼 준비
     - **E4 — 건강 / sensitive (GDPR Art.9 유사)**: KR 의료법 + EU GDPR Art.9 + Quebec Bill 25 reference. consent: separate granular opt-in per data class (medication / cycle / symptom / mental). retention: stricter delete + portable export. examples: 난임, 항암, 만성질환, 정신건강
   - **각 asset record 의 metadata 에 박히는 fields**: `envelope`, `consent_version_id`, `data_classes[]`, `retention_policy_id`, `export_eligibility`
   - **Validation pipeline**: spawn 시 LLM envelope hint → server-side validator (deterministic 규칙 + keyword 매칭) → mismatch 시 human-readable explanation 으로 사용자에게 confirm

9. **Adaptive Engine (ADR-0010 4-layer cache)**
   - L1 device + L2 hint + L3 edge + L4 cron
   - per-user $0.11 → $0.04 sublinear cost target

10. **Instrumentation (P4 운영 지표 + thesis falsification 도구)**
    - D1/D7/D30 retention per asset
    - 2nd-asset-creation rate (horizontality 의 단일 시그널)
    - month-3 memory-retrieval-rate (>30day-old reads) — P4 의 measured truth
    - `asset_authoring_time_internal` (P2 internal 검증, target <1시간)
    - `asset_spawn_time_user` (UX 검증, target <2분)
    - contribution margin per asset (cost: LLM + push + storage; revenue: subscription tier)
    - PostHog opt-in only (ADR-0005 준수)

**Suggested internal sequencing (모든 항목 ship, 단계만 다름)**:
- **Sprint 1 (Day 1-3)**: ADR-0004 폐기 + 의존 문서 rewrite + Supabase/R2 setup + Edge Function (Anthropic + Whisper proxy)
- **Sprint 2 (Day 4-7)**: Asset Spawner core (LLM → asset schema → preset rendering) + onboarding 확장 (voice/photo) + ChapterMemory write/read
- **Sprint 3 (Day 8-10)**: Multi-asset switcher polish + TPO routing (Calendar + Geo) + Push notification system
- **Sprint 4 (Day 11-12)**: Compliance matrix + Privacy/Consent layer + Adaptive engine cache + Instrumentation
- **Sprint 5 (Day 13)**: Polish (Pretendard/Fraunces fonts, 다크 모드, accessibilityLabels) + Jest preset + EAS Build

## Open Questions

- **ADR-0004 폐기의 blast radius**: CLAUDE.md (`v1 출시 스코프`, `v1 카테고리 단속`), grammar.md (난임-specific 예시), Phase 1 event schema (cycle_*, embryo_*) 모두 horizontal pivot 에 맞춰 재정렬 필요. **Sprint 1 Day 1-2 의 deliverable** 로 본 design scope 내에서 atomic commit 시리즈로 처리 (별도 세션 분리 안 함 — Dependencies 의 Blocker 도 Sprint 1 으로 흡수).
- **App Store category**: fertility/health 에서 "Lifestyle" 또는 "Productivity" 로 변경 검토. KR App Store 의 카테고리 선택이 discoverability 에 직결.
- **Korean tagline**: "MODU" + "Listen to your life" 유지 가능. 그러나 마케팅 카피의 fertility 단서 ("먼저 IVF 동반자로 시작") 제거 필요.
- **Pricing tier 재정의**: horizontal 에서도 ₩5,900 / ₩11,900 가 맞는지. multi-asset 에서 Family tier 의 의미 — asset-per-member? 또는 unlimited assets? Plus tier 도 asset 개수 제한 vs unlimited 결정 필요.
- **B2G consent UX 노출 시점**: hidden surface 라도 onboarding 에서 plain 한 disclosure 필요. 어느 시점·어느 화면에서 expose 할지.
- **Fertility asset 의 위치**: warm seed 로만 두는지, 아니면 onboarding flow 의 첫 preset 선택지로도 노출하는지. preset 으로 노출하면 fertility 사용자 acquisition 이 빠르지만 horizontal 메시지가 흐려질 수 있음.

## Success Criteria

**Timeline anchor**: Sprint Day 14 = TestFlight release. 모든 in-market metric 은 release 이후 시점부터 카운트.

**Day 14-21 (release → +7d, internal testing wave)**:
- `asset_spawn_time_user` <2분 (median across all installs)
- `asset_authoring_time_internal` <1시간 (founder/팀 measured, P2 검증)
- crash rate <1%, ANR <0.5%
- 첫 5개 vertical 에서 각각 ≥1 install + 1st-asset 완료

**Day 14-28 (release → +14d, in-market window)**:
- D7 retention ≥30% (전체 install)
- 2nd-asset-creation rate ≥15% (horizontality falsifier; <5% → P1 재검토 trigger)
- 유저 NPS ≥30 (in-app survey, opt-in)

## Reviewer Concerns

독립 spec-reviewer (대화 맥락 미공개) 가 본 doc 에 대해 4/10 quality 평가를 내렸고, 그 중 일부는 founder 지시 ("skip X 금지, 되는 방향으로만") 와 구조적 충돌 — 따라서 fix 가 아니라 **명시적 acknowledged risk** 로 박아둠.

**RC1 — 14일 sprint 안의 ALL-IN 10개 항목 전체 production-quality ship 이 mathematically tight.** Reviewer 산출: 1.4일/항목 (push + voice + multi-asset + vector DB + 4-layer cache + compliance matrix + ChapterMemory sync-ready + instrumentation + privacy/consent + LLM asset gen). Mitigation 은 R2/R4 (lock-screen 위젯, L2/L4 cache 는 release 후 OTA — Day 14 = "TestFlight 가능" 시점 정의로 sequencing). 이 acknowledged risk 가 받아들여지지 않으면 ALL-IN scope 의 일부를 v1.1 로 명시 강등 필요 — founder 결정 사항.

**RC2 — LLM→preset 렌더 quality 의 prompt iteration 비용이 budget 안 잡힘.** Reviewer 추정: 5-20 prompt 반복 + eval harness. 본 doc 은 Sprint 1 의 30개 seed eval harness 로 흡수했으나 (R1), 첫 release 의 LLM 출력 품질이 사용자 첫인상에 직결되는 risk 는 잔존. Mitigation 효과 측정은 in-market 이후 (Day 14-21 의 NPS).

**RC3 — Vector DB 의 정부 정책 corpus 는 "data sourcing 자체가 프로젝트".** Reviewer 의 지적: 정부24 + 부처별 sourcing 은 weeks of work. 본 doc 은 fertility/study/childcare 3개 vertical 만 v1 ingest 로 축소 (R3) — 나머지 vertical 의 P 축은 "정보 준비 중" placeholder. 이 자체가 사용자 경험에서 inconsistency 노출 risk.

**RC4 — Adaptive engine 4-layer cache (ADR-0010) 의 from-scratch 설계는 14일에 fantasy.** 본 doc 은 L1+L3 만 v1, L2+L4 는 OTA 로 분리 (R4). 다만 "ADR-0010 후보 → committed v1" 으로 promote 한 것 자체가 위험 surface — release 후 architecture 가 4-layer compatible 인지 단계적 검증 필요.

**RC5 — Lock-screen widget 의 native binary 의존성 (factual constraint).** iOS WidgetKit / Android AppWidget 은 OTA 로 추가 불가능 — store review 가 필요한 native binary 작업. v1 binary 에는 minimal widget (오늘의 nudge 1개 표시) 만 포함, multi-asset switcher 가 가능한 풍부한 widget 은 v1.1 binary 로 명시 분리. (R2 mitigation 의 factual correction)
