The hardest question

Why didn't this render?

A PM asks why a campaign banner isn't showing. A QA engineer can't reproduce a missing CTA. You add a console.log, check the feature flag dashboard, re-read the condition logic — and still can't point to one authoritative answer.

The rules ran. The output came back. But the trace only tells you what was selected.

The gap

LaunchDarkly, OpenFeature, and XState are built around the selected path. They return a winner. They don't output a structured record of what was excluded, what was outcompeted, or what the runtime decided it lacked enough context to resolve.

1

Every suppressed component carries a whyNot reason code — condition mismatch, policy gate, or confidence shortfall. Nothing is silently dropped.

2

Locked slots are named primitives, not inferred from priority ties. A locked slot carries an explicit reason field and wins against policy suppress.

3

Abstentions — cases where context fell below a confidence threshold — are explicit trace entries, not silent drops. The gap is named, not ignored.

4

The result is a single output that a developer can diff, a QA engineer can assert against, and a PM can read without needing to understand the rule engine internals.

What others miss

The full decision surface

Other tools return a winner. TPO returns the entire decision — selected, suppressed, locked, and deferred — with a structured reason for each outcome.

How it works

evaluate(context, options) → result

A pure, deterministic function. Same inputs always produce the same output. No network calls, no side effects, no hidden state.

Input

context
TPOContext
stage, role, time.phase, place, locale, device, meta
rules
RulePack
Versioned set of rules: conditions, priority, confidenceHint per component
lockedSlots
LockedSlots
Operator-pinned slots with named reason. Resolved before rule evaluation.
policy
PolicyPack
Compliance / business rules that suppress components post-evaluation
confidenceThreshold
number
Default 0.60. Below this, a matched rule abstains rather than selects.
evaluate() ↓

Output — TPOResult

selected
SelectedComponent[]
Winning components with slot, specificity, priority, matched rule IDs
suppressed
SuppressedComponent[]
Excluded components with human-readable whyNot and optional policyId
locked
LockedSlot[]
Operator-pinned slots. Cannot be suppressed by policy.
abstentions
AbstainedComponent[]
Confidence below threshold — named with gap, not silently dropped
riskTier
'low' | 'medium' | 'high'
Derived from policy hits, abstentions, and locked overrides in this pass
quickstart.ts TypeScript 
import { evaluate, formatTrace } from '@tpo/runtime';
import type {
  TPOContext, EvalOptions
} from '@tpo/runtime';

const context: TPOContext = {
  stage: 'cart-abandonment',
  role:  'returning-buyer',
  time:  { phase: 'flash-sale' },
  device: 'mobile',
};

const options: EvalOptions = {
  rules: {
    version: '1.0.0',
    rules: [
      {
        id: 'r-coupon',
        componentKey: 'CartDiscountCoupon',
        slot: 'primary_cta',
        conditions: [
          { field: 'stage', op: 'eq',
            value: 'cart-abandonment' },
          { field: 'time.phase', op: 'eq',
            value: 'flash-sale' },
        ],
        priority: 1,
        confidenceHint: 0.92,
      },
      {
        id: 'r-review',
        componentKey: 'WriteReviewPrompt',
        slot: 'nudge',
        conditions: [
          { field: 'stage', op: 'eq',
            value: 'post-purchase' }, // won't match
        ],
        priority: 2,
      },
    ],
  },
  confidenceThreshold: 0.60,
};

const result = evaluate(context, options);

// result.selected[0].componentKey  → 'CartDiscountCoupon'
// result.suppressed[0].componentKey → 'WriteReviewPrompt'
// result.suppressed[0].whyNot
//   → 'stage eq "post-purchase" — got "cart-abandonment"'

console.log(formatTrace(result, context, {
  mode: 'governance',
}));

Risk tiers

Decision confidence, quantified.

riskTier is derived automatically from the count of policy hits and abstentions in a single evaluation pass. Use it for downstream gating, alerting, or audit logging.

low

No policy gates triggered, at most one abstention, and at least one component was selected or locked. The runtime resolved all slots cleanly against the rule set.

policyHits === 0
abstentions <= 1
selected.length > 0 or locked.length > 0
medium

At least one policy gate triggered, or two or more components abstained due to insufficient confidence. Human review recommended before promotion.

policyHits >= 1
— or —
abstentions >= 2
high

Two or more policy gates triggered, three or more abstentions, or no output at all (no selected and no locked components). Treat as a signal to block downstream rendering or escalate for review.

policyHits >= 2
— or — abstentions >= 3
— or — selected.length === 0 && locked.length === 0

Get started

Up and running in 5 minutes.

No account required. No configuration server. A pure TypeScript function you can run in any environment — Node, Bun, Deno, edge runtimes.

terminal
npm install @tpo/runtime
first-trace.ts TypeScript 
import {
  evaluate, formatTrace
} from '@tpo/runtime';

// Or use a domain starter pack:
import {
  bankingContext,
  bankingOptions,
} from '@tpo/runtime/starters/banking';

const result = evaluate(
  bankingContext,
  bankingOptions,
);

console.log(
  formatTrace(result, bankingContext, {
    mode: 'governance',
  })
);
